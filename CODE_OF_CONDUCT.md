# Code of Conduct

Status: Draft
Version: 0.1
Project: RedOps Framework

---

## Purpose

RedOps Framework is a community project for authorized pentest and red team workflow design.

This Code of Conduct defines expected behavior for contributors, maintainers, reviewers, and community participants.

The goal is to keep discussion professional, useful, safe, and aligned with authorized security work.

---

## Scope

This Code of Conduct applies to:

* GitHub issues
* Pull requests
* Discussions
* Reviews
* Documentation contributions
* Example files
* Templates
* Project-related communication channels
* Maintainer communication
* Security reports

It also applies when someone represents the project in a public or community context.

---

## Expected Behavior

Participants are expected to:

* Be professional and respectful.
* Keep discussion focused on the project.
* Provide clear technical feedback.
* Use specific examples when reviewing documentation or schemas.
* Respect different experience levels.
* Accept constructive review.
* Avoid unnecessary argument.
* Avoid personal attacks.
* Keep security work inside authorized scope.
* Report sensitive issues through the correct channel.
* Remove private or sensitive data before posting.

---

## Unacceptable Behavior

The following behavior is not accepted:

* Harassment
* Threats
* Personal attacks
* Insults
* Discriminatory language
* Sexualized language or imagery
* Public sharing of private information
* Posting credentials, tokens, private keys, or secrets
* Posting real client data
* Posting private logs or internal network data
* Encouraging unauthorized access
* Encouraging credential theft
* Encouraging malware deployment
* Encouraging evasion or bypass activity
* Encouraging destructive activity
* Repeated off-topic comments
* Spam
* Misrepresentation of project purpose

---

## Security-Specific Rules

RedOps Framework is intended for authorized security work.

Community contributions must not include:

* Unauthorized exploitation instructions
* Credential dumping procedures
* Malware code
* Payload delivery workflow
* EDR or AV bypass instructions
* Phishing delivery workflow
* Destructive test instructions
* Persistence payloads for unauthorized systems
* Real client data
* Private logs
* Internal IP ranges from real environments
* Credentials, tokens, or private keys

High-risk content may be edited, rejected, or removed.

Maintainers may convert unsafe content into safer documentation, abstract examples, or policy notes.

---

## Contribution Standards

Good contributions should be:

* Clear
* Specific
* Reviewable
* Maintainable
* Aligned with authorized use
* Consistent with the project safety model
* Written in professional technical English

Good contribution areas include:

* Documentation improvement
* Safety model improvement
* Data model improvement
* Schema proposal
* Example file improvement
* Template improvement
* ATT&CK mapping model
* Telemetry model
* Report workflow
* LLM assistance boundary
* API proposal

---

## Review Standards

Review comments should focus on:

* Accuracy
* Safety
* Scope alignment
* Readability
* Maintainability
* Missing assumptions
* Missing limitations
* Inconsistent terminology
* Broken links
* Incorrect schema structure
* Risk of misuse

Review comments should not attack the contributor.

---

## Enforcement

Maintainers may take action when this Code of Conduct is violated.

Possible actions include:

1. Asking for clarification.
2. Editing or removing unsafe content.
3. Closing an issue or pull request.
4. Warning the participant.
5. Temporarily limiting participation.
6. Blocking a participant from the project.
7. Reporting serious abuse to the relevant platform.

Enforcement should be based on the behavior, risk, and impact.

---

## Maintainer Responsibilities

Maintainers are responsible for:

* Applying this Code of Conduct consistently.
* Reviewing reports in good faith.
* Protecting private reports.
* Removing unsafe content when needed.
* Keeping project discussion aligned with authorized use.
* Explaining moderation decisions when appropriate.
* Avoiding unnecessary public exposure of sensitive information.

Maintainers may reject contributions that create legal, ethical, operational, or misuse risk.

---

## Public Issues and Pull Requests

Public issues and pull requests should not contain sensitive data.

Before posting, remove:

* Credentials
* Tokens
* Private keys
* Internal hostnames
* Internal IP addresses
* Private URLs
* Client names without permission
* Sensitive logs
* Proprietary data
* Personal data

Use safe examples such as:

```text
example.com
app.example.com
10.10.10.0/24
project-001
asset-001
evidence-001
```

---

## Project Alignment

All community activity should align with the project purpose:

```text
Authorized pentest and red team workflow management.
```

The project does not support unauthorized activity, malware development, credential theft, evasion, destructive testing, or out-of-scope operation.
